# 本地开发工作流程

## 📌 工作原则

1. **本地优先开发**：所有小优化和修复先在本地进行
2. **线上稳定优先**：线上版本基本不动，只有确认稳定后才更新
3. **充分测试后推送**：上传前必须确保功能正常

---

## 🔄 标准工作流程

### 第一阶段：本地开发

1. **修改代码**
   - 直接编辑本地文件
   - 遵循编码规范（参见 new_rules.txt）

2. **本地测试**
   ```bash
   # 方法1：直接在浏览器打开 index-new.html
   # 方法2：使用本地服务器（如果有）
   ```

3. **验证功能**
   - ✅ 清除浏览器缓存（Ctrl+Shift+R）
   - ✅ 测试修改的功能
   - ✅ 检查控制台无错误
   - ✅ 检查移动端显示

### 第二阶段：决定是否推送

**推送条件**（满足任一即可）：
- 有重大功能更新
- 修复了影响使用的bug
- 有重要的安全更新
- 定期维护（如每周）

**不推送条件**：
- 仅本地测试用途的临时修改
- 未完成的开发中功能
- 可能破坏现有功能的实验性修改

### 第三阶段：推送到线上

1. **更新版本号**
   ```javascript
   // 在 index-new.html 中更新
   版本: 2026-06-01.6  →  2026-06-01.7
   ```

2. **提交代码**
   ```bash
   git add .
   git commit -m "描述你的修改"
   ```

3. **推送到线上**
   ```bash
   # 推送到 gh-pages（会自动部署到线上）
   git push origin gh-pages
   
   # 同步到 main 分支
   git checkout main
   git merge gh-pages
   git push origin main
   git checkout gh-pages
   ```

4. **验证线上**
   - 访问线上版本
   - 强制刷新（Ctrl+Shift+R）
   - 确认功能正常

---

## 📊 分支策略

| 分支 | 用途 | 说明 |
|------|------|------|
| `gh-pages` | 线上部署分支 | 自动部署到 GitHub Pages |
| `main` | 源代码主分支 | 与 gh-pages 保持同步 |
| `dev` | 开发分支（可选） | 保留本地开发记录 |

---

## ⚠️ 注意事项

### 不要做的事
- ❌ 不要频繁推送到线上（每次推送会影响所有用户）
- ❌ 不要在未测试的情况下推送
- ❌ 不要推送会导致数据丢失的修改
- ❌ 不要使用 PowerShell 批量替换

### 必须做的事
- ✅ 每次修改前查看 Git 历史
- ✅ 修改后先本地测试
- ✅ 推送前确认功能正常
- ✅ 更新版本号
- ✅ 遵循编码规范

---

## 🔧 紧急修复流程

如果线上出现严重bug需要紧急修复：

1. **立即修复**
   - 在本地快速修复
   - 充分测试

2. **快速推送**
   - 更新版本号
   - 提交并推送
   - 通知用户刷新页面
