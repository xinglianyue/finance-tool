编程规范强制约束

一、流程强制要求

每次编程任务必须遵循：
1. 需求重述 - 明确目标、约束、输出、成功标准
2. 需求澄清 - 技术栈未指定时使用 AskUserQuestion
3. 任务规划 - 创建 TodoWrite 分解任务
4. 分步执行 - 按计划执行，实时更新状态
5. 验证交付 - 检查清单 + 安全提醒

二、绝对不能做的事（最高优先级 - 已有惨痛教训）

1. 绝对不能使用 PowerShell 批量替换修改 HTML/JS 文件！
   - 问题描述：使用 (Get-Content -Raw) -replace '旧内容', '新内容' | Set-Content 批量替换会破坏 UTF-8 编码，导致中文乱码、页面显示空白或崩溃。已经发生过 2 次！（2026-05-30）
   - 替代方案：
     - 使用 Edit 工具（小范围精确修改）
     - 使用 Python（Python 脚本精确处理 UTF-8 编码）
     - 使用 Git（先恢复正确版本，再小心修改）
     - 单文件小改动：直接用 Edit 工具，不使用任何批量替换

2. 不能在不清空浏览器缓存的情况下测试修复
   - 问题描述：用户看到的还是旧版本代码，测试结果不准确，已经发生过多次
   - 替代方案：
     - 强制刷新：Ctrl+Shift+R（Windows）/ Cmd+Shift+R（Mac）
     - 清除缓存：Ctrl+Shift+Delete
     - 添加版本号参数：script.js?v=20260530

3. 不能忽视 CORS 跨域问题
   - 问题描述：GitHub Pages 无法通过 fetch 访问 raw.githubusercontent.com，导致云端数据加载失败（2026-06-01）
   - 替代方案：
     - 将数据文件放在同一域名下
     - 使用相对路径加载（如 ./shared-data.json）
     - 配置 CORS 代理（备选）

三、必须避免的事（高优先级）

4. 不能假设数据结构，必须先调试确认
   - 教训来源：2026-05-30 修复数据显示问题
   - 错误地假设 merchantData.all?.cities，实际应该是 merchantData.cities，导致 currentData.cities 为空数组
   - 正确做法：
     - 先加 console.log() 打印实际数据结构
     - 确认路径后再修改代码
     - 不要盲目猜测

5. 不能在没有备份的情况下修改文件
   - 教训来源：多次文件损坏经历
   - 修改前必须检查 Git 历史
   - 确保可以快速回退
   - 或者先复制一个备份文件

6. 不能忽视数据格式兼容性
   - 教训来源：2026-06-01 数据加载失败
   - 云端数据格式为旧格式 { date, cities, fileName }，代码期望新格式 { all: {...}, ka: {...}, city: {...} }
   - 正确做法：
     - 添加格式检测逻辑
     - 自动转换旧格式为新格式
     - 保持向后兼容性

四、必须始终做的事（标准流程）

7. 每次修改前：先看 Git 历史，确认最后正确版本
   git log --oneline -10  # 查看最近提交
   git show <commit>      # 查看某次提交的具体修改

8. 每次修复后：先在本地完全测试，再推送到线上
   本地测试步骤：
   1. 修改代码
   2. 打开本地文件测试
   3. 清除浏览器缓存
   4. 确认功能正常
   5. 检查控制台无错误
   6. 再提交和推送

9. 版本号管理
   - 每次功能修改：更新版本号
   - 版本格式：YYYY-MM-DD.N （N 为当天修改次数）
   - 确保所有人看到的是同一版本
   - 更新位置：页面显示版本号 + JS 文件版本参数

10. 防御性编程
    - 所有数据访问必须有空值保护
    - 使用 ?. 可选链操作符
    - 使用 || 或 ?? 设置默认值
    - 添加数据格式检测和边界检查

五、代码质量强制要求

- 每个函数必须有 JSDoc 注释（@param、@returns、@description）
- 所有 async 操作必须有 try-catch
- 输入必须验证（空值、类型、长度、格式）
- 禁止硬编码凭证（密码、API Key、Token）
- SQL 必须参数化查询（? 占位符）
- 密码必须 bcrypt 哈希
- HTML 输出必须转义（防止 XSS）
- 单文件不超过 300 行
- 禁止使用 var、substr()、alert()
- 使用 let 或 const 替代 var
- 使用 showToast 替代 alert

六、禁止行为

- 返回假数据/空壳实现（必须标注 TODO）
- setInterval 无清理机制
- 硬编码端口/URL
- 数学方法返回字符串（.toFixed()）
- CSS 类名重复定义
- import 不存在的模块
- 忽视浏览器缓存问题
- 忽视数据格式兼容性

七、问题解决流程标准

遇到问题时的标准步骤：

1. 确认问题
   - 详细描述问题
   - 截图和控制台日志
   - 清除浏览器缓存后重新确认

2. 检查 Git
   - 查看当前状态
   - 确认最后工作版本
   - 可以立即回退

3. 调试数据
   - 加 console.log 打印实际数据
   - 确认数据结构
   - 不要盲目猜测

4. 小范围修改
   - 只改问题所在的那一行
   - 使用 Edit 工具，不用批量替换
   - 确保 UTF-8 编码正确

5. 本地测试
   - 先在本地完全测试
   - 清除缓存
   - 确认所有功能正常

6. 推送到线上
   - 提交修改
   - 推送
   - 确认线上也正常

八、工具使用指南

场景                  推荐工具           避免使用
-------------------------------------------------
小范围修改1-2行        Edit工具           PowerShell批量替换
精确到行号修改          Python脚本         PowerShell
恢复正确版本            git checkout       手动重写
查看实际数据结构        console.log()      猜测
测试修改                本地打开文件测试   直接推送到线上
数据加载                相对路径           跨域URL

九、分支管理规范

- main 分支：源代码主分支，用于开发和维护
- gh-pages 分支：GitHub Pages 部署分支
- 确保 shared-data.json 同步到 gh-pages 分支
- 每次部署必须递增版本号

十、维护阶段规范

- 定期清理未跟踪的临时脚本和测试文件（每周）
- 保持文档与代码同步（每月）
- 确认数据备份完整（每月）
- 检查页面加载速度和内存使用（按需）

十一、交付时必须附带

- 代码质量检查结果
- 安全提醒（依赖验证、漏洞检查）
- .gitignore 配置说明
- 版本号更新记录

十二、调试指南

1. 日志策略
   - 使用统一的日志管理
   - console.log：正常业务日志
   - console.warn：警告信息
   - console.error：错误信息

2. 诊断面板
   - 使用内置诊断面板收集调试信息
   - 包含：浏览器信息、本地存储状态、云端数据状态、控制台日志

3. 数据检查
   - console.log(JSON.stringify(data, null, 2))
   - 使用 typeof 检查类型
   - 检查数组长度
